# 都バス Dify デモ実装計画

## 目的

都バス公式ホームページ提案に向けて、開発陣が「Dify を入れると何ができるか」を具体的に想像できる最低限のデモを作る。

このデモでは、完成度の高い本番アプリではなく、以下を見せることを優先する。

- 地図上でリアルタイム車両位置が動くこと
- 右側のチャットから地図や運行情報へつながること
- 将来的に Dify、FAQ、GTFS/GTFS-RT、公式データを接続できる構造であること

## 現在の判断

このリポジトリを継続利用する。

理由:

- 左に地図、右にチャットという入れ物が提案イメージに合っている
- FastAPI がフロントエンド静的配信と API をまとめて扱っており、PoC の速度が出る
- 既に GTFS-Realtime 風の取得・整形レイヤーがあり、都バスへの差し替えに向いている
- 別リポジトリを作ると、器づくりからやり直しになる

ただし、現在の NYC MTA デモを壊さないように、新しいブランチで進める。

推奨ブランチ:

```bash
git switch -c feature/toei-bus-demo
```

## 調査メモ

都バスのリアルタイムデータは公開されている。

公共交通オープンデータセンターのカタログに、東京都交通局の都バスについて **GTFS-RT 形式のバス関連リアルタイム情報** が掲載されている。形式は Protocol Buffers、ライセンスは CC BY 4.0 とされている。

また、静的なバス関連情報も GTFS/GTFS-JP として公開されている。

参考:

- 公共交通オープンデータセンター概要: https://www.odpt.org/overview/
- 都バス GTFS-RT リアルタイム情報: https://ckan.odpt.org/dataset/b_bus_gtfs_rt-toei
- 都バス GTFS/GTFS-JP 静的情報: https://ckan.odpt.org/dataset/b_bus_gtfs_jp-toei
- 公共交通オープンデータチャレンジ 2025 エントリー案内: https://challenge2025.odpt.org/ja/entry.html

## 開発者登録について

公共交通オープンデータセンターの利用には、開発者サイトへのユーザ登録が必要になる可能性が高い。

調査時点では、公共交通オープンデータチャレンジの案内に以下の趣旨の記載がある。

- 公共交通オープンデータセンターに開発者登録する
- 登録内容の確認に最大 2 営業日程度かかる
- 個人、グループ、法人での参加に関する記載がある

そのため、個人の学習・検証目的での登録は自然な使い方に見える。ただし、実際の提案利用、クレジット表記、API キーの扱い、利用条件は登録後に開発者サイトと各データセットのライセンスを確認する。

開発者サイト:

https://developer.odpt.org/signup

## 想定アーキテクチャ

```text
Frontend
  - Leaflet 地図
  - 都バス車両マーカー
  - 停留所・路線表示
  - Dify 風チャット UI

FastAPI Backend
  - 都バス静的 GTFS/GTFS-JP 取得・キャッシュ
  - 都バス GTFS-RT VehiclePosition 取得・整形
  - Dify チャットプロキシ
  - 地図操作コマンドの受け渡し

Dify
  - ユーザー質問の解釈
  - FAQ/RAG 回答
  - FastAPI API 呼び出し
  - map_command 生成

ODPT / Public Transport Open Data Center
  - GTFS/GTFS-JP 静的情報
  - GTFS-Realtime VehiclePosition
```

## 実装フェーズ

### Phase 0: 現状デモの安定化

完了済み:

- ローカル起動確認
- Docker がない環境でも Python 仮想環境で起動可能なことを確認
- Leaflet CDN 依存をローカル同梱に変更
- 地図初期化失敗時もチャットが初期化されるように修正

残タスク:

- ローカル起動手順を README または別 docs に追記
- 起動中プロセスとログファイルの扱いを整理

### Phase 1: 都バス風デモへの最小差し替え

目的:

ODPT 登録前でも、開発陣に見せられる都バス向けの画面イメージを作る。

作業:

- アプリ名と文言を都バス向けに変更
- 地図初期位置を東京都心に変更
- MTA 駅データを都バス停留所モックに差し替え
- MTA 車両モックを都バス車両モックに差し替え
- 路線カラー、凡例、チャット例を都バス向けに変更
- API の外形はなるべく維持し、後で実データに差し替えやすくする

この段階の成功条件:

- 地図上に都バス風の停留所と車両が表示される
- チャットから「都庁前を表示して」のような操作イメージが伝わる
- 実データ接続前でも提案用の画面として説明できる

### Phase 2: 都バス GTFS-RT VehiclePosition 接続

ステータス: 初期接続済み

目的:

リアルタイムに動くことを主役にする。

作業:

- `.env` に ODPT 用の設定を追加: 完了
- GTFS-RT VehiclePosition を取得するサービスを追加: 完了
- Protocol Buffers をパースしてフロント向け JSON に整形: 完了
- 車両 ID、路線 ID、緯度経度、タイムスタンプを表示: 完了
- 取得失敗時はモックにフォールバックする: 完了
- GTFS内部の route_id と利用者向け系統名の対応付け: 未対応

この段階の成功条件:

- 都バスの実車両位置が地図に表示される: 完了
- 更新ボタンまたは定期ポーリングで位置が更新される: 完了
- データ取得失敗時もデモ画面が壊れない: 完了

接続先:

- 公開URL: https://api-public.odpt.org/api/v4/gtfs/realtime/ToeiBus
- 登録後URL: https://api.odpt.org/api/v4/gtfs/realtime/ToeiBus?acl:consumerKey=[アクセストークン]

ローカルメモ:

- このPCではPython/httpxの証明書検証で失敗したため、`.env` に `ODPT_SSL_VERIFY=false` を設定して検証している
- 本番や共有デモ環境では `ODPT_SSL_VERIFY=true` を原則にする

### Phase 3: 都バス GTFS/GTFS-JP 静的データ接続

目的:

停留所・路線・系統名などの基本情報を実データに近づける。

作業:

- GTFS/GTFS-JP の取得方法を確認
- stops/routes/trips/shapes を必要最小限で読み込む
- 停留所検索 API を追加
- 車両位置と路線・停留所情報を紐づける
- 地図上で停留所、路線、車両の関係が分かるようにする

この段階の成功条件:

- 実データ由来の停留所が表示される
- チャットやクリック操作で停留所情報を参照できる
- 車両がどの系統か分かる

### Phase 4: Dify 接続

目的:

「単なるチャット」ではなく、公式情報、地図、リアルタイムデータをつなぐ対話レイヤーとして見せる。

作業:

- Dify API キーを `.env` で設定
- 既存 `/api/chat/message` を活用
- 地図操作コマンドを Dify レスポンスから受け取る
- Dify から FastAPI の検索 API を呼べる構成を検討
- FAQ/RAG 用の想定文書を用意

優先するデモ質問:

- 「今、都庁前付近を走っているバスを見せて」
- 「東京駅から近い都バス停を教えて」
- 「このバスはどの方面に向かっていますか」
- 「車いすで利用しやすい停留所を教えて」
- 「忘れ物をした場合はどこに問い合わせればいいですか」

この段階の成功条件:

- Dify が回答だけでなく地図操作につながる
- API や FAQ を根拠にしている構造が説明できる
- 「ここを本番データベースや公式 FAQ に接続すればよい」と開発陣が理解できる

## 実装方針

- まずはリアルタイム車両位置を最優先にする
- Dify は最後に足す
- データ取得層は MTA と都バスを切り替えられるように分離する
- API キーやトークンは `.env` に置き、コミットしない
- 実データが取得できない場合でもデモが壊れないよう、モックフォールバックを残す
- クレジット表記は画面または README に明示する

## 注意点

- GTFS-RT の URL、認証方式、トークン要否は登録後に開発者サイトで確認する
- CC BY 4.0 のクレジット表記が必要
- 本番提案で見せる場合は、データ提供元、取得間隔、遅延、正確性の但し書きを入れる
- Dify 単体ではデータベース接続にならないため、FastAPI 側の検索 API または Dify Tool 連携が必要
- 「リアルタイムに動く」体験を優先し、FAQ/RAG は後段にする
